# 장바구니 쿠폰 적용 기능 — 요구사항 명세 (v1.0)

> 학습용 PM 스펙. 실 결제(PG), 인증, 백엔드 API는 범위 외이며 페이지 로컬 상태로만 동작한다.

## 1. 배경
demo-web 의 장바구니 페이지에 결제 직전 단계의 **쿠폰 적용** 기능을 추가한다.

## 2. 범위 (Scope)
- 대상 페이지: `apps/demo-web/cart.html`
- 사용자: 로그인 여부 무관, 모두 동일 동작
- "결제하기" 클릭 시 동작은 "결제 완료 모달" 노출까지 (실제 결제 X)

## 3. 기능 명세

### F-1. 장바구니 표시
- 페이지 진입 시 다음 3종이 초기 상품으로 표시된다.

  | 상품 ID | 상품명  | 단가     | 초기 수량 |
  |---------|---------|----------|----------|
  | P-001   | 티셔츠  | 12,000원 | 1        |
  | P-002   | 머그컵  | 8,500원  | 2        |
  | P-003   | 에코백  | 15,000원 | 1        |

- 각 상품 행에는 상품명, 단가, 수량 (`-` / `+`), 라인 합계, "행 삭제" 버튼이 노출된다.
- 라인 합계 = 단가 × 수량
- 수량은 **1 이상 99 이하**의 정수

### F-2. 합계 계산
- **상품 금액 합계** = 모든 라인 합계의 합
- **할인 금액** = 적용된 쿠폰에 의해 산정 (F-3 참조), 적용된 쿠폰이 없으면 0
- **최종 결제 금액** = 상품 금액 합계 − 할인 금액

### F-3. 쿠폰
보유 쿠폰 목록은 페이지에 다음과 같이 미리 표시된다.

| 코드      | 종류 | 값    | 최소 주문 금액 | 최대 할인 한도 | 유효기간              |
|-----------|------|-------|----------------|----------------|-----------------------|
| FLAT5000  | 정액 | 5,000원 | 30,000원       | —              | ~2099-12-31           |
| RATE10    | 정률 | 10%   | 20,000원       | 10,000원       | ~2099-12-31           |
| RATE20    | 정률 | 20%   | 50,000원       | —              | ~2024-12-31 (만료됨) |

#### 적용 규칙
- 한 번에 **하나의 쿠폰만** 적용할 수 있다.
- 사용자는 보유 쿠폰 목록에서 라디오 형태로 하나를 선택한 뒤 "쿠폰 적용" 버튼을 눌러 적용한다.
- 적용된 쿠폰은 "쿠폰 해제" 버튼으로 해제할 수 있다.
- 적용 불가능한 쿠폰(최소 주문 금액 미충족 / 만료)은 목록에서 **비활성화 상태**로 표시되며 선택 자체가 불가능하다.

#### 할인 계산
- 정액 쿠폰: `할인 금액 = 쿠폰 값`
- 정률 쿠폰: `할인 금액 = 상품 금액 합계 × 비율`. 최대 할인 한도가 있으면 그 한도를 초과할 수 없다.

### F-4. 화면 동작
- 수량 변경 / 상품 삭제 / 쿠폰 적용·해제 시 합계가 **즉시** 갱신된다.
- 쿠폰을 적용한 뒤 장바구니가 변경되어 해당 쿠폰의 적용 조건을 더 이상 충족하지 않게 되면, 쿠폰은 **자동 해제**되고 안내가 노출된다.

### F-5. 결제
- "결제하기" 버튼 클릭 시 결제 완료 모달이 노출된다 (실 결제 없음).
- 장바구니가 비어 있을 때 결제 버튼은 비활성화 상태이다.

## 4. 비기능 요구사항
- 데스크탑 Chrome 최신 버전 기준으로 동작한다.

## 5. 분석 과제 (요구사항 분석 → TC → 자동화로 이어짐)

### 5.1. 1차 분석 (R · 요구사항 분석)
[`assignment/requirements/`](../../../../QA-Portfolio/assignment/requirements/README.md) 양식 그대로 5가지 기준으로 분석한다.
1. 사용자
2. 주요 기능
3. 상태 변화
4. 예외 상황
5. 외부 시스템 또는 정책 조건

### 5.2. 모호점 / 누락 도출
이 명세를 그대로 구현하려고 하면 **개발자가 결정을 강요받는 지점**이 여러 곳 있다.
명세를 다시 읽으며 다음 형식으로 정리해 본다.

```
| # | 모호한 지점 | 왜 모호한가 | 내가 세운 가정 |
```

> 힌트: F-2 의 "최종 결제 금액"이 음수가 될 수 있는지, F-3 의 할인 계산 결과가 **소수**가 나올 때 어떻게 처리할지, 쿠폰이 **1회용인지 재사용 가능**한지, 동일 쿠폰을 적용한 채로 페이지를 새로고침하면 상태가 유지되는지… 이런 질문이 떠올라야 한다. (의도적으로 비워둔 부분이 더 있다 — 찾아라.)
