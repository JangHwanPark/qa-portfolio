# 3단계 · 테스트 케이스 도출 (모범답안)

[02_scenarios.md](02_scenarios.md) 의 시나리오를 동등 분할 / 경계값 기법으로 TC 표로 풀어낸 예시.

## 컬럼 정의
- **유형**: Positive / Negative / Boundary / Negative-Boundary
- **우선순위**: P0(필수) / P1(중요) / P2(보통) / P3(낮음)
- **비고**: 연결되는 SC ID 명시

## 작성 조건 (참고)
- **최소 15개 이상** TC
- 동등 분할 클래스(활성 / 잠금 / 휴면 / 미가입) 모두 커버
- 경계값 명시적 표기: 실패 횟수 4 / 5 / 6, 비번 길이 7 / 8 / 20 / 21
- 한 TC = 한 검증 포인트 (한 단계에서 여러 단정 금지)
- 기대 결과는 단정 가능한 표현 (모호 표현 금지)

## 학습 포인트

- **경계값은 4개씩 묶기**: 임계값 N에 대해 `N-1 / N / N+1` 만 잡으면 부족. 비번 길이는 `최소-1, 최소, 최대, 최대+1` 네 점(7/8/20/21)을 모두 잡아야 양 끝의 동등 분할 클래스가 갈린다.
- **한 TC = 한 단정**: TC-004 는 "카운터 = 5 AND 잠금 전환 AND 메시지 노출" 세 가지를 한 번에 본다고 적었지만, 엄격히는 메시지/상태/카운터 별로 쪼개는 게 회귀 디버깅에 유리. 학습 단계에선 묶어도 되지만 실제 도구(TestRail 등)에선 분리 권장.
- **사전 조건 명시**: TC-003 의 "실패 카운터 = 3" 같이 **테스트 시작 시점의 시스템 상태**를 적어야 자동화에서 사전 데이터 세팅 전략이 결정된다. 자동화 단계에서 이게 곧 "API seed vs UI 5회 클릭" 분기가 된다.
- **사용자 열거 방지**: TC-002 와 TC-009 의 기대 메시지가 **완전히 동일** 해야 함. 메시지가 다르면 보안 결함. 이런 항목은 TC 에 정확한 텍스트를 박아두는 게 좋다.
