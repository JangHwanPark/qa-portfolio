# 2단계 · 시나리오 설계 / 우선순위 (모범답안)

[01_requirements.md](01_requirements.md) 분석 결과를 출시 전 검증 시나리오로 풀어낸 예시.

## 컬럼 정의
- **우선순위**: P0(머스트, 출시 차단) / P1(중요) / P2(보통) / P3(여유 시)
- **분류**: Smoke / Sanity / Regression / Exploratory / Performance / Security

## 작성 조건 (참고)
- **최소 10개 이상** 시나리오
- 정책 임계값(잠금 5회 / 30분, 휴면 90일) 검증 시나리오 포함
- 비기능 시나리오(보안 / 접근성) **최소 1개 이상**
- 우선순위 분포 가이드: P0 20% 내외, P1 30~40%, P2 30%, P3 10~20%

## 회귀 영역 포함 메모
- 로그인은 인증 의존 기능의 진입점이므로 SC-01 은 모든 빌드 Smoke 에 필수
- 잠금 / 휴면 정책은 보안 회귀 대상으로 nightly Regression 에 고정 편성
- 비밀번호 재설정 메일은 외부 시스템 의존이므로 자동 검증은 발송 트리거까지만

## 학습 포인트

- **P0 은 "출시 차단" 수준만**: 정상 로그인(SC-01)과 잠금 정책(SC-02, SC-03)에만 P0 부여. "중요해 보이니까 P0" 가 아니라 "깨지면 출시 불가능한가" 로 판단한다.
- **시나리오는 TC 가 아니다**: "5회 실패 후 잠금"은 한 시나리오. 그 안에 "4회는 잠금 아님", "5회째 잠금 전환", "6회는 카운터 추가 증가 없음" 같은 TC 가 풀려 나온다. SC 와 TC 의 입도 차이.
- **비기능을 빠뜨리지 않기**: SC-07(Security), SC-12(Accessibility) 처럼 기능 외 시나리오를 의도적으로 포함. 평가자가 가장 자주 확인하는 항목.
- **시간 의존 시나리오 표시**: SC-05 의 "(시계 조작)" 같이 자동화 난도가 보이도록. 4단계에서 자동/부분자동/제외 분류와 직결된다.
